## Technical Implementation Plan

### Phase 2.1: Extend Template Style System

#### Update TemplateStyle Interface
Add new properties to `src/lib/resume/types.ts`:
```typescript
interface TemplateStyle {
  // ... existing properties ...
  
  // Layout Structure
  layoutType: 'single-column' | 'two-column' | 'three-column' | 'two-column-header' | 'two-column-sidebar';
  leftColumnWidth?: number;    // Percentage for column layouts
  rightColumnWidth?: number;
  middleColumnWidth?: number;
  
  // Date Format
  dateFormat: 'full' | 'month-year' | 'year-only' | 'month-year-short' | 'numeric' | 'custom';
  dateSeparator?: string;      // e.g., "-", "/", "."
  showPresent?: boolean;        // Show "Present" vs "Current"
  
  // Bullet Style
  bulletStyle: 'round' | 'square' | 'dash' | 'diamond' | 'arrow' | 'custom';
  bulletColor?: string;        // Color for custom bullets
  bulletCharacter?: string;    // Custom bullet character
  
  // Section Grouping
  sidebarSections?: string[];  // Sections to show in sidebar
  mainSections?: string[];     // Sections to show in main area
}
```

## ATS Compatibility Requirements

### Critical Rules:
1. **Columns**: Use tables or CSS that converts to linear text in DOCX
2. **Dates**: Always use standard date formats (no icons/graphics)
3. **Bullets**: Use standard Unicode characters, not images
4. **Fonts**: Fallback to standard fonts in DOCX
5. **Structure**: Maintain semantic HTML structure
6. **Headers**: Use standard section headers (SUMMARY, EXPERIENCE, etc.)
7. **Text**: No images, graphics, or complex shapes
8. **Layout**: Ensure text flows logically when parsed linearly

### Column Layout Strategy:
- For DOCX: Use Word tables (ATS-friendly)
- For preview: Use CSS Grid (visual only)
- Ensure linear text flow when parsed
